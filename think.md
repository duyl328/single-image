# Single Image 目标与规划

## 总目标

这个项目的核心目标，是为大规模照片库提供一套本地优先的筛选工作流，先解决重复与相似照片清理问题，再逐步扩展到人工评分、AI 辅助评分、精筛和归档整理。

当前照片规模预估可达到数百个文件夹、总量 800GB 以上，因此系统设计必须优先考虑：

- 增量扫描
- 大规模吞吐
- 低误删风险
- 可中断、可恢复
- 人工决策优先于自动删除

## 分阶段目标

### 第一步：重复与相似照片检测

目标：

- 检测完全重复照片
- 检测高相似照片
- 支持大库增量扫描
- 优先帮助用户快速精简图库

要求：

- 完全重复检测必须高准确率
- 相似检测宁可多给候选，也不能高误报
- 扫描过程不能每次全库重算
- 第一次扫描慢可以接受，但后续扫描必须明显加速
- 删除动作必须谨慎，默认以“候选 + 人工确认”为主

当前判断：

- 这一阶段是整个产品的基础
- 在这一步做稳之前，不应继续快速堆后续复杂功能
- 当前实现最大问题不是 UI，而是扫描吞吐、增量策略和相似分组质量

### 第二步：人工评分与快捷键工作流

目标：

- 提供简洁的照片预览界面
- 支持快捷键 `0-5` 快速打分
- 支持快速浏览、跳过、删除、保留
- 评分可以后续同步到 EXIF 或 XMP

设计原则：

- 评分操作应先写入本地数据库
- 不要每次按键都立即写 EXIF，避免高 I/O 和格式兼容问题
- EXIF/XMP 写回应作为独立批处理能力
- 用户评分与系统技术评分必须分离

### 第三步：AI 学习用户审美并辅助评分

目标：

- 基于用户已经评分过的照片，学习用户偏好
- 为未评分照片生成预测分数
- 帮助用户优先处理值得保留的照片

设计原则：

- 第一版只做“建议分数”，不做自动删除
- 应输出预测分数和置信度
- 更适合先学习“相近照片中你更喜欢哪张”，再学习绝对 1-5 分
- 这一阶段必须建立在已有足够人工评分数据基础上

预期使用方式：

- 1-2 分照片优先折叠或标记为低优先级
- 3-5 分照片进入人工复核流程
- AI 负责排序和减轻工作量，不直接替用户做不可逆决策

### 第四步：精筛流程

目标：

- 对候选优质照片做快速精筛
- 提供单张模式和双张/多张对比模式
- 支持快速归类、删除、保留
- 提高人工筛片效率

设计原则：

- 必须围绕键盘操作效率设计
- 必须支持预加载和低延迟切换
- 删除和归类应支持撤销
- 精筛应消费前面阶段已有的分组、评分和 AI 预测结果

### 第五步：归类保存

目标：

- 对最终保留的优质照片进行分类整理
- 支持例如“个人照片”“相机照片”等归档方案
- 后续可能涉及大规模 I/O 和目录重组

当前状态：

- 这一步暂时不作为近期重点
- 等前四步稳定后再设计数据迁移和归档方案

## 当前优先级

近期只聚焦前四步，其中优先级如下：

1. 第一步：重复与相似照片检测
2. 第二步：人工评分与快捷键
3. 第四步：精筛流程
4. 第三步：AI 评分

原因：

- 第一步决定整个系统是否可用
- 第二步和第四步直接决定人工处理效率
- 第三步依赖前两步积累的数据闭环

## 当前产品原则

### 原则 1：先做可靠，再做聪明

在没有稳定增量扫描、低误报相似检测和可靠人工复核流程之前，不要急于加入更复杂的 AI 功能。

### 原则 2：自动化以“建议”为主，不以“直接删除”为主

尤其是相似照片、RAW/JPEG 配对、AI 评分等，都不应直接触发不可逆操作。

### 原则 3：所有重任务都应可增量、可中断、可恢复

面对 800GB 级图库，任何需要全量重跑的设计最终都会失效。

### 原则 4：评分体系必须分层

至少应区分：

- 用户人工评分
- 技术质量分
- AI 预测分

三者不能混为一个字段。

### 原则 5：I/O 要谨慎

缩略图、特征、元数据写回、文件移动、归档整理都可能是瓶颈，后续设计时必须把 I/O 成本当成一等约束。

## 下一阶段建议

接下来优先输出一份更具体的设计草图，内容包括：

- 扫描与分析任务拆分
- 增量索引策略
- 重复与相似的结果模型
- 人工评分的数据模型
- 精筛页面的操作流
- AI 评分前需要积累的数据

## 一句话总结

短期先把“能高效、可靠地找出重复和相似照片”做好；中期做“人工评分 + 精筛”；后期再做“AI 学习审美并辅助排序”。

## 工程化补充思考

这一节专门回答几个关键问题：

- 如何做到可中断、可后台运行
- 文件移动后是否还能正确识别
- 如何避免每次对 800GB 全量跑 SHA-256
- 大量记录下 SQLite 如何稳定写入
- 非图片文件是否要纳入体系
- 缩略图和 RAW 的处理策略如何更合理

### 1. 可中断、可后台运行怎么做

结论：

- 不需要要求照片“不能挪位置”
- 也不需要每次从头扫到尾
- 关键在于把扫描拆成可恢复的阶段任务，而不是做成一次性同步大任务

建议把扫描拆成以下阶段：

1. 发现文件
2. 采集元数据
3. 计算 quick hash
4. 计算完整内容哈希
5. 提取视觉特征
6. 生成缩略图
7. 生成重复/相似候选

每个文件都应记录自己的处理状态，例如：

- `discovered`
- `metadata_ready`
- `quick_ready`
- `sha_ready`
- `visual_ready`
- `thumb_ready`
- `grouped`
- `failed`

这样做的好处：

- 扫描进行到一半关闭应用，下次可以继续
- 后台任务可以按阶段慢慢跑
- 可以优先完成“最值钱”的阶段，例如先出完全重复结果，再慢慢补相似分析
- 新增文件只进入未完成阶段，不影响已完成的大库结果

后台运行的基本方式：

- 前台只负责创建扫描任务和展示进度
- 后台维护一个任务队列
- Worker 从队列中取文件处理
- 每完成一个阶段就把状态写入数据库
- 用户点击停止时，只是把任务标记为 `cancelling`
- Worker 在每个阶段边界检查取消标记，安全停下

这就是“可中断、可后台运行”的核心，不是靠单次扫描更快，而是靠任务拆分和状态持久化。

### 2. 文件被移动了怎么办

结论：

- 不应要求用户扫描后不能移动照片
- 系统应尽量做到“路径变化不影响内容身份判断”

路径变化的识别建议分三层：

#### 第一层：路径未变

如果路径相同，且：

- 文件大小相同
- 修改时间相同
- 文件系统身份相同

则直接跳过，不做重算。

#### 第二层：同卷移动或重命名

在 Windows 上可以使用：

- 卷序列号
- 文件 ID

如果文件 ID 相同，只是路径变了，就把它视为“同一个文件换了位置”，不需要重新分析内容。

这也是当前项目已有基础能力的方向。

#### 第三层：跨卷移动、复制、重新导入

跨盘移动后，文件 ID 会变化，这时不能再依赖文件系统身份。

解决方式：

- 用 quick hash + 文件大小 + 旧记录状态做候选匹配
- 只有命中候选时才进一步计算完整 SHA-256
- 如果最终内容哈希相同，则视为同内容新实例或位置变化

所以，用户可以移动照片，系统不需要禁止，只需要做到“最终一致”。

### 3. 如何避免 800GB 全量跑 SHA-256

这是核心问题。

首先要澄清：

- 跑 SHA-256 不会把 800GB 一次性读进内存
- 但确实会把 800GB 从磁盘完整读一遍
- 真正的瓶颈是磁盘 I/O，而不是内存

因此，大库扫描必须是分层判定，而不是“新文件先无脑全哈希”。

推荐的精确重复检测流水线：

#### 第 0 层：目录发现

先只拿：

- 路径
- 扩展名
- 文件大小
- 修改时间
- 文件系统身份

这一层很便宜。

#### 第 1 层：仅对新增或疑似变化文件计算 quick hash

quick hash 可以取：

- 文件头部切片
- 中间切片
- 尾部切片
- 再拼接文件大小

这样只需要读取几百 KB，而不是整个文件。

#### 第 2 层：按 `文件大小 + quick hash` 建桶

只有桶里出现冲突时，才值得继续跑完整 SHA-256。

也就是说：

- 文件大小唯一：不需要立刻跑 SHA-256
- 同大小但 quick hash 唯一：也通常可以先不跑
- 同大小且 quick hash 冲突：再升级为完整 SHA-256

这一步会极大降低磁盘全量顺序读的压力。

#### 第 3 层：仅对“需要视觉分析”的图片做解码

精确重复和相似分析要拆开：

- 精确重复靠元数据 + quick hash + SHA-256
- 相似分析靠缩略图或视觉特征

不要为了相似分析而强迫所有文件一开始就解码图片。

#### 第 4 层：相似分析只处理候选集

不要全库两两比。

应该先通过：

- 扩展名类型
- 分辨率范围
- 拍摄时间邻近
- pHash/dHash 或其他粗特征

先缩小候选集，再做较贵的 SSIM 或其他相似度计算。

### 4. Quick Hash 是不是应该作为主入口

是的。

对于你的场景，合理顺序应该是：

1. metadata
2. quick hash
3. full hash
4. visual features

而不是：

1. metadata
2. full hash
3. quick hash
4. visual features

quick hash 的主要价值不是“替代 SHA-256 做最终判定”，而是：

- 决定是否值得升级到完整内容哈希
- 支撑大规模增量扫描
- 显著减少无意义的全文件读取

因此：

- quick hash 适合做初筛
- SHA-256 适合做最终确认

两者不是竞争关系，而是分层关系。

### 5. SQLite 会不会扛不住

结论：

- 十万级、几十万级记录对 SQLite 不是问题
- 真正的问题是“多线程同时小写入”而不是“记录数量”

SQLite 的正确使用方式应该是：

- 多 worker 并行做文件读取和分析
- 只有一个 writer 线程持有写连接
- worker 通过 channel 把结果提交给 writer
- writer 批量写入数据库

推荐策略：

- 打开 WAL 模式
- 使用单独写连接
- 一批写入一个事务
- 重复使用 prepared statements
- 避免每处理一个文件就单独 commit

不建议：

- 多线程同时直接往同一个 SQLite 写
- 每个文件单独开事务
- 把数据库写入和图像解码耦合在同一线程里

所以答案是：

- SQLite 可以继续用
- 但必须采用“多 worker 分析 + 单 writer 批量落库”的模式

### 6. 非图片文件要不要考虑

要，但要分层处理。

不是所有文件都应该进入同一套视觉分析链路。

建议分成几类：

#### A. 主照片文件

例如：

- jpg
- jpeg
- png
- webp
- heic
- heif
- raw

这是当前核心处理对象。

#### B. 视频文件

例如：

- mp4
- mov

第一阶段建议：

- 先纳入目录索引
- 支持 metadata + quick hash + exact duplicate
- 不急于做视频相似

这样不会把系统复杂度一下拉爆。

#### C. Sidecar 文件

例如：

- aae
- xmp

这些文件不应被当成普通照片来独立清理，而应尽量与主照片关联：

- 跟随原图
- 作为原图附属信息
- 删除时联动提醒

否则会误删编辑记录。

#### D. 压缩包和其他文件

例如：

- zip
- rar
- 7z

第一阶段建议：

- 记录存在
- 默认不展开内容
- 不进入照片分析主链路
- 仅在“未知格式/非媒体文件”面板中展示

所以不是“都要处理”，而是“都要感知，但分层处理”。

### 7. 缩略图从 PNG 改成 JPG 是否更好

大方向上是更好的。

对于照片缩略图：

- PNG 文件通常更大
- 写入和读取成本通常更高
- 对照片内容并不划算

更合理的建议是：

- UI 预览缩略图用 JPEG
- 质量控制在合适范围，例如 80-85
- 对于透明背景图片再保留 PNG 特例

但有一个细节要注意：

当前系统把缩略图不仅用于 UI，还用于相似分析。

因此更合理的长期方案不是简单把 PNG 改成 JPG，而是分离两类缓存：

- UI 缩略图缓存
- 相似分析特征缓存

也就是说：

- UI 缩略图可以是 JPEG
- 相似分析最好直接缓存灰度小图或视觉特征
- 不要让“给人看”的缩略图格式反过来限制算法实现

### 8. RAW 文件如何处理更合理

你的判断是对的：

- RAW 不应默认走完整解码
- 更合理的是优先提取内嵌预览图

推荐策略：

#### 对 RAW 的精确重复检测

直接使用：

- metadata
- quick hash
- full hash

不需要解码 RAW。

#### 对 RAW 的预览和相似分析

优先使用：

- 内嵌 JPEG 预览
- 厂商生成的 preview image

只有在以下情况才考虑完整 RAW 解码：

- 用户主动打开大图预览
- 没有可用内嵌预览
- 后续有明确高质量渲染需求

这样可以大幅减少 CPU 和 I/O 开销。

### 9. 真正适合大图库的扫描策略

如果图库未来真的稳定到数百目录、800GB 以上，仅靠“每次遍历所有目录 + stat + 判断”也会逐渐显得重。

因此后续可以考虑两层机制：

#### 基础层：定期全量校准

例如：

- 手动全量扫描
- 每隔较长周期做一次全量校验

#### 增量层：基于文件系统变化记录

在 Windows 上，可以进一步研究：

- 文件系统 watcher
- NTFS USN Journal

这样就可以只处理真正变化过的文件，而不是每次重新遍历整个大库。

这不是第一版必须做，但如果目标真的是长期管理超大图库，这是很值得的方向。

## 更新后的第一阶段设计原则

第一阶段不再只是“把重复和相似找出来”，而是要满足以下工程约束：

- 支持可中断、可恢复
- 支持后台长期运行
- 支持用户在扫描过程中移动文件
- 先用 metadata + quick hash 降低 I/O
- 只在必要时计算完整 SHA-256
- 只对候选图片做视觉解码
- SQLite 使用单 writer 批量写入
- 非图片文件按类型分层处理
- RAW 优先提取内嵌预览
- UI 缩略图缓存与相似分析缓存应分离
